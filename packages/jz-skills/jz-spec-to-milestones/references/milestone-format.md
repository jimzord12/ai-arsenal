# Milestone plan format

The milestone plan is a single fenced `## Milestones` section appended to `SPEC.md`. The HTML-comment fence lets a re-run find and replace it in place — never leave two blocks.

## Output block

```markdown
<!-- milestone-plan:start -->

## Milestones

> Milestone plan for <spec_id>. Each milestone is a demoable, end-to-end increment; its slug becomes the `Milestone:` tag on the issues stage 3 writes. Generated by `jz-spec-to-milestones`.

### 1. <Milestone title> — `<milestone-slug>`

- **DependsOn:** none
- **Decomposed:** pending
- **Demo:** <one sentence>

### 1. <Title> — `<slug>`

- **Demo:** <one sentence — what someone does and sees that proves this works end-to-end>
- **DependsOn:** none | `<slug>`[, `<slug>`]
- **Delivers:** FR-00x, FR-00y; NFR-00z; DEC-00x, SUBDEC-00y
- **Carries:** QUE-00x (default: <the SPEC's default, condensed>) | none
- **Scope:** <1–3 sentences: the end-to-end path this lights up. No file paths — scoping files is stage 4/5.>

### 2. <Title> — `<slug>`

- **Demo:** …
- **DependsOn:** `<slug>`
- **Delivers:** …
- **Carries:** none
- **Scope:** …

### Coverage

| Requirement             | Milestone         |
| ----------------------- | ----------------- |
| FR-001                  | `<slug>`          |
| FR-002                  | `<slug>`          |
| FR-014 (send button)    | `explicit-send`   |
| FR-014 (clear + gating) | `manage-and-gate` |
| NFR-001                 | cross-cutting     |

<!-- milestone-plan:end -->
```

Rules:

- The heading number is display order only; the **slug** is the identifier (`DependsOn` and the coverage table reference slugs, not numbers). One identifier, no drift.
- Slugs are kebab-case and tag-friendly: `capture-and-store`, `explicit-send`, `manage-and-gate`.
- Every new milestone starts with `Decomposed: pending`. Preserve an existing canonical ISO timestamp when regenerating an unchanged slug.
- `Delivers` lists the requirement ids this milestone makes demoable, plus the decision ids it enacts. The `Coverage` table is the exhaustive check — every `FR-`/`NFR-` in the SPEC appears there. Most appear once. Two appear more than once, each explicitly: a `cross-cutting` NFR, and a **bundled FR** whose several demoable behaviours split across milestones (name the part in parentheses). Nothing is unassigned or silently dropped.
- No file paths anywhere in the plan. Milestones describe demoable behaviour; files get scoped by `jz-issue-to-contract` (stage 4).

## Worked example (illustrative)

A "share a photo" feature with `FR-001` upload, `FR-002` list, `FR-003` delete, `FR-004` thumbnail, `FR-005` caption, `NFR-001` images encrypted at rest.

**Horizontal (wrong)** — these demo nothing alone:

- `storage-layer` — the DB tables and blob store.
- `api-layer` — the endpoints.
- `ui-layer` — the screens.

**Demoable (right):**

### 1. Upload and see one photo — `upload-one-photo`

- **Demo:** pick a photo, upload it, see it appear with its thumbnail.
- **DependsOn:** none
- **Delivers:** FR-001, FR-004; NFR-001
- **Carries:** none
- **Scope:** the thinnest end-to-end path — capture, encrypted store, thumbnail, render one image. The walking skeleton every later milestone widens.

### 2. Browse the gallery — `browse-gallery`

- **Demo:** open the gallery and scroll a grid of uploaded photos with captions.
- **DependsOn:** `upload-one-photo`
- **Delivers:** FR-002, FR-005
- **Carries:** none
- **Scope:** list many stored photos and their captions; needs photos to exist, so it follows the skeleton.

### 3. Delete a photo — `delete-photo`

- **Demo:** long-press a photo, confirm, watch it disappear from the gallery.
- **DependsOn:** `browse-gallery`
- **Delivers:** FR-003
- **Carries:** none
- **Scope:** removal from store and grid.

`NFR-001` is `cross-cutting` for storage-touching milestones but is first _delivered_ by the skeleton, so it is listed there and marked cross-cutting in the coverage table.

If the feature were just upload + view with no delete or captions, the right answer is **one** milestone (`upload-and-view`, `DependsOn: none`) delivering everything — don't split for ceremony.
